# 2.0 - Discovery and Current Industry Gaps

This document outlines the current landscape of AI-assisted software development and identifies key gaps that GarageBuild is positioned to fill.

## 1. The "Wrapper" vs. "Platform" Gap
**Current State:** Most prominent tools in the market (Cursor, Windsurf) are highly optimized IDE wrappers around specific LLM APIs (OpenAI/Claude). They win on UX polish and provide a premium chat experience.
**The Gap:** These tools do not manage the *infrastructure* of development. They can write code, but they don't inherently understand or manage Docker lifecycles, complex deployment pipelines, or multi-provider cost optimization as an integrated system.
**GarageBuild Opportunity:** Be the platform that manages the entire engineering environment—not just a chat box inside a text editor.

## 2. Model Lock-in & Volatility
**Current State:** Platforms like v0.dev and Google AI Studio are heavily biased toward specific providers (Vercel/Next.js, Google Gemini). This creates "Platform Lock-in."
**The Gap:** If an AI provider changes their pricing model, deprecates a specific version of an LLM, or suffers from cloud outages, the developer's workflow is disrupted. There is no easy way to rotate models without changing code or prompt structure.
**GarageBuild Opportunity:** By treating AI Providers as **hot-swappable plugins**, GarageBuild ensures that software remains independent of any single infrastructure provider.

## 3. Contextual Fragmentation (The "Context Stuffing" Problem)
**Current State:** Developers currently copy/paste files into a chat window or upload folders to an AI Studio, hoping it understands the context.
**The Gap:** A folder doesn't convey *execution context*. An LLM can see a file, but it can't easily see that the local server is throwing a 504 error because of a specific SQLite constraint in a related package unless that information is manually piped through.
**GarageBuild Opportunity:** The **Engine Layer** provides persistent access to the workspace, database state, and logs as context, rather than relying on manual uploads from the user.

## 4. Ephemeral Prototyping vs. Permanent Engineering
**Current State:** Tools like Replit Agent are excellent for "disposable" apps—apps that exist in a cloud container and never need to be migrated or managed locally.
**The Gap:** Real engineering projects require **standardized software.** They must live on local drives, use standard package managers (npm/pip), allow multi-developer collaboration via Git, and survive the retirement of the tool that helped build them.
**GarageBuild Opportunity:** Focus on "Software Permanence." Every GarageBuild action results in standardized source code that is independent of the platform's existence a month from now.

## 5. Enterprise Privacy & Local Sovereignty
**Current State:** High-performance AI requires cloud access, but many industries (Finance, Healthcare, Defense) are barred from sending data to these clouds.
**The Gap:** Local options like Ollama exist but lack the orchestration layer needed to manage a multi-model project where some tasks *can* be offloaded and others *must* stay local for compliance.
**GarageBuild Opportunity:** Provide a unified interface that manages local/cloud split seamlessly, allowing organizations to define "Compliance zones" within their projects.

## 6. The Infrastructure Gap (The Gap Between Code & Cloud)
**Current State:** There is currently no tool that provides a seamless transition from *Prompt $\rightarrow \text{Code Generation}$ $\rightarrow \text{Infrastructure Deployment}$.* Users must manually set up Nginx, move files to Docker, and configure ports.
**The Gap:** Rapid prototyping ends when the code "works on my machine" but fails in production due to lack of infra-configuration.
**GarageBuild Opportunity:** Integration of **Deployment Plugins** into the core workflow ensures that a prototype moves to production using standardized infrastructure (like Docker) from day one.
